\doxysubsection{Detailed Description}
Entry point for the Firebase \doxylink{classfirebase_1_1firestore_1_1_firestore}{Firestore} C++ SDK. 

To use the SDK, call \doxylink{classfirebase_1_1firestore_1_1_firestore_a3ad8553b7861a52a9b69847ce59c85f9}{firebase\+::firestore\+::\+Firestore\+::\+Get\+Instance()} to obtain an instance of \doxylink{classfirebase_1_1firestore_1_1_firestore}{Firestore}, then use \doxylink{classfirebase_1_1firestore_1_1_firestore_ad63645003fc293ca2adadee0c451ae20}{Collection()} or \doxylink{classfirebase_1_1firestore_1_1_firestore_a63e7155867ce6daf744c1738e029492a}{Document()} to obtain references to child paths within the database. From there, you can set data via \doxylink{classfirebase_1_1firestore_1_1_collection_reference_ad67c20ff97e0b6b91b869badeca12013}{Collection\+Reference\+::\+Add()} and \doxylink{classfirebase_1_1firestore_1_1_document_reference_ad4a7cf6a1e9a51cafd99d3f9aeaeaeb7}{Document\+Reference\+::\+Set()}, or get data via \doxylink{classfirebase_1_1firestore_1_1_query_a207650647a8a4668c5f759afe27c7273}{Collection\+Reference\+::\+Get()} and \doxylink{classfirebase_1_1firestore_1_1_document_reference_a1b665ad5f842a184222c14fde179a252}{Document\+Reference\+::\+Get()}, attach listeners, and more.

\begin{DoxyNote}{Note}
\doxylink{classfirebase_1_1firestore_1_1_firestore}{Firestore} classes are not meant to be subclassed except for use in test mocks. Subclassing is not supported in production code and new SDK releases may break code that does so. 
\end{DoxyNote}

Attaches a listener for a snapshots-\/in-\/sync event. Server-\/generated updates and local changes can affect multiple snapshot listeners. The snapshots-\/in-\/sync event indicates that all listeners affected by a given change have fired.

NOTE\+: The snapshots-\/in-\/sync event only indicates that listeners are in sync with each other, but does not relate to whether those snapshots are in sync with the server. Use {\ttfamily \doxylink{classfirebase_1_1firestore_1_1_snapshot_metadata}{Snapshot\+Metadata}} in the individual listeners to determine if a snapshot is from the cache or the server.

Waits until all currently pending writes for the active user have been acknowledged by the backend.

The returned future is resolved immediately without error if there are no outstanding writes. Otherwise, the future is resolved when all previously issued writes (including those written in a previous app session) have been acknowledged by the backend. The future does not wait for writes that were added after the method is called. If you wish to wait for additional writes, you have to call {\ttfamily Wait\+For\+Pending\+Writes} again.

Any outstanding {\ttfamily Wait\+For\+Pending\+Writes} futures are resolved with an error during user change.
